# 图片上传与预览复制功能优化复盘报告

## 1. 问题背景

在排版系统的实际使用过程中，发现了两个严重影响用户体验的核心问题：

1.  **“复制预览”格式污染**：用户在 HTTP 环境下点击“复制预览”并粘贴到微信编辑器时，HTML 源代码中的换行和缩进会被浏览器转义为大量的 `&nbsp;`（不换行空格），且存在加粗等样式丢失的问题。
2.  **图片批量上传“死锁”与失败**：用户上传多张图片时，往往只有第一张成功，后续图片会卡住或报错“网络原因”，上传速度极慢，且并发处理逻辑存在漏洞导致队列假死。

## 2. 根因分析

### 2.1 复制污染根源
- **浏览器转义行为**：系统在复制前没有对 HTML 进行彻底压缩。浏览器在执行复制动作时，会将源代码中的格式化空白（换行、Tab）视为视觉布局的一部分，为了在粘贴时保留这种布局，会自动将其转换为 `&nbsp;`。
- **DOM 渲染损耗**：旧逻辑依赖于浏览器重新解析 HTML 字符串进行复制，这一过程会导致部分内联样式被过滤或修改。

### 2.2 上传失败根源
- **前端 Slot 释放漏洞**：`UploadManager` 在任务失败进入重试延迟时，没有立即释放并发槽位（Slot），导致槽位被占满后无法加载新任务，产生“假死”。
- **后端 Token 刷新冲突**：当并发数 > 1 时，若 Access Token 正好过期，多个上传请求会同时触发 Token 刷新动作。微信端对高频刷新有限制，且无锁状态下的并发刷新会导致 Token 状态不一致，使后续请求失效。
- **接口选型性能差**：原先使用微信“永久素材”接口（`add_material`），该接口在微信内部逻辑重、速度慢，在高并发下极易超时。

## 3. 解决方案实施

### 3.1 预览复制逻辑重构
- **极致 HTML 压缩 (Minification)**：在将 HTML 写入剪贴板前，通过正则彻底移除所有换行、制表符，并将标签间的空格完全清理。
- **事件拦截策略 (Copy Event Hook)**：不通过浏览器默认渲染路径，而是直接监听 `copy` 事件，手动将原始、无损的 HTML 数据写入 `clipboardData`，彻底杜绝浏览器二次处理。

### 3.2 上传流水线全链路加固
- **前端 `UploadManager` 优化**：
    - 确保 `activeUploads` 计数在单次上传动作（无论成败）结束后**立即释放**。
    - 引入更清晰的控制台日志系统（[BATCH_START], [SUCCESS], [RELEASE]），让用户实时感知并发状态。
- **后端微信服务重构**：
    - **Token 并发锁 (Promise Locking)**：在服务端实现 Token 刷新的并发锁。当一个请求正在刷新时，后续请求会静默排队等待结果，确保全过程只有一个刷新请求。
    - **切换为 CDN 极速接口**：将上传接口更换为 `media/uploadimg`。该接口专门用于文章插图，不占配额且响应时间从“秒级”缩短为“毫秒级”。
- **全链路超时对齐**：将前端、API 拦截器及后端微信请求的超时时间统一延长至 60s，增强在不稳定网络下的抗干扰能力。

## 4. 优化结果验证

1.  **格式一致性**：粘贴到微信编辑器的内容不再包含冗余空格，样式（加粗、颜色）完整保留。
2.  **高并发稳定性**：支持多张图片（8+ 张）同时并发上传，Token 刷新点不再报错。
3.  **上传速度**：单张图片上传耗时大幅下降，整体批量处理效率提升约 300%。

## 5. 经验总结

- **针对微信生态的特殊处理**：微信编辑器对外部粘贴的 HTML 极其敏感，必须在源码级别做到极致压缩。
- **锁机制的重要性**：在处理具备状态（如 Token）的公共资源时，必须在后端建立保护层，由于 Access Token 的刷新是全局敏感操作，单点锁是解决大规模并发异常的关键。
- **全链路监控**：在复杂的文件上传流中，从前端槽位到后端代理再到微信 API，每一环的超时和日志记录必须对齐，才能在出现“网络错误”时快速定位。
