# 系统级架构设计文档

## 一、架构概览

### 1.1 系统目标
- 轻量级元数据管理系统
- 零图片存储（图片由微信托管）
- 低成本运营（月成本 < ¥50）
- 高可用性（99.9%）
- 易于维护和扩展

### 1.2 核心架构原则
1. **单一职责原则** - 每个服务只负责一个明确的功能域
2. **最简代码原则** - 不做向后兼容，保持代码最简化
3. **渐进式演进** - 从简单开始，根据需求逐步扩展
4. **数据分离原则** - 元数据与文件存储分离

### 1.3 系统边界
```
┌─────────────────────────────────────────────────────────────┐
│                        系统边界                              │
├─────────────────────────────────────────────────────────────┤
│  输入：用户操作、Webhook事件、定时任务                        │
│  输出：飞书表格更新、统计数据、通知消息                        │
│  存储：JSON文件（元数据）、微信（图片）                       │
│  处理：内容管理、同步触发、数据转换                           │
└─────────────────────────────────────────────────────────────┘
```

## 三、核心组件设计

### 3.1 Webhook 接收层
```yaml
组件职责:
  - 接收来自前端的Webhook事件
  - 验证请求合法性
  - 将事件推送到处理队列

设计要点:
  - 幂等性保证（使用event_id防重）
  - 快速响应（<100ms）
  - 事件分类路由

关键接口:
  - POST /webhook/content
  - POST /webhook/image
  - POST /webhook/publish
```

### 3.2 事件处理层
```yaml
组件职责:
  - 从队列消费事件
  - 业务逻辑处理
  - 触发飞书同步

设计要点:
  - 异步处理
  - 错误重试机制
  - 批量处理优化

处理器类型:
  - ContentCreatedHandler
  - ContentUpdatedHandler
  - ImagesUploadedHandler
  - ContentPublishedHandler
```

### 3.3 存储层
```yaml
存储策略:
  - 元数据：JSON文件（结构化存储）
  - 索引：内存缓存 + 定期持久化
  - 配置：JSON文件

文件结构:
  /data/
    ├── metadata/          # 元数据文件
    │   ├── 2024/
    │   │   ├── 01/
    │   │   │   └── contents.json
    │   │   └── 02/
    │   └── index/
    │       ├── all.json   # 全量索引
    │       └── latest.json # 最新更新
    ├── config/           # 配置文件
    │   ├── sync.json     # 同步配置
    │   └── mapping.json  # 字段映射
    └── logs/             # 日志文件
        ├── webhook/
        └── sync/
```

### 3.4 飞书集成层
```yaml
组件职责:
  - 管理飞书连接器配置
  - 处理数据同步
  - 错误处理和重试

同步策略:
  - 增量同步（基于时间戳）
  - 批量同步（100条/批）
  - 失败重试（3次，指数退避）

API封装:
  - TableMetaService（表结构）
  - RecordsService（记录操作）
  - SyncService（同步管理）
```

## 四、数据模型设计

### 4.1 核心实体
```yaml
Content（内容）:
  id: string                # 唯一标识
  title: string             # 标题
  editor: string           # 编辑者
  source: string           # 图文来源
  author: string           # 文案作者
  category: string         # 分类
  tags: string[]           # 标签列表
  status: enum             # 状态(draft/published)

  # 微信相关
  wechat_images: WechatImage[]  # 微信图片列表
  wechat_draft_id: string?      # 微信草稿ID

  # 时间信息
  created_at: timestamp        # 创建时间
  updated_at: timestamp        # 更新时间
  published_at: timestamp?     # 发布时间

  # 统计信息
  view_count: number           # 浏览次数
  like_count: number           # 点赞数

  # 内容
  summary: string              # 摘要(前200字)
  content_url: string?         # 内容详情URL

WechatImage（微信图片）:
  id: string               # 图片ID
  media_id: string         # 微信媒体ID
  url: string              # 图片URL
  name: string             # 文件名
  size: number             # 文件大小
  status: enum             # 状态
  error_msg: string?       # 错误信息
```

### 4.2 飞书表格映射
```yaml
飞书表格字段映射:
  - id: content_id         # 内容ID（主键）
  - title: title           # 标题
  - editor: editor         # 编辑者
  - source: source         # 图文来源
  - author: author         # 文案作者
  - category: category     # 分类（单选）
  - status: status         # 状态（单选）
  - created_at: created_at # 创建时间
  - updated_at: updated_at # 更新时间
  - image_count: image_count  # 图片数量
  - view_count: view_count    # 浏览次数
  - summary: summary          # 内容摘要
  - detail_url: content_url   # 详情链接
```

## 五、关键流程设计

### 5.1 内容创建流程
```yaml
步骤:
  1. 前端收集用户输入
  2. 上传图片到微信（获取URL）
  3. 发送Webhook事件
  4. 后端保存元数据到JSON
  5. 更新内存索引
  6. 触发飞书同步
  7. 返回成功响应

性能指标:
  - 响应时间: < 500ms
  - 可用性: 99.9%
  - 并发支持: 100 QPS
```

### 5.2 图片处理流程
```yaml
步骤:
  1. 前端选择本地图片
  2. 调用微信上传API
  3. 获取media_id和URL
  4. 保存图片信息到元数据
  5. 触发内容更新事件

优化点:
  - 并行上传多张图片
  - 压缩大图片
  - 上传进度反馈
```

### 5.3 飞书同步流程
```yaml
触发条件:
  - 内容创建/更新
  - 定时批量同步
  - 手动触发

执行步骤:
  1. 检查飞书连接状态
  2. 获取需要同步的数据
  3. 批量转换数据格式
  4. 调用飞书API更新
  5. 记录同步日志
  6. 处理失败重试

监控指标:
  - 同步成功率 > 99%
  - 延迟 < 5分钟
  - API调用次数 < 1000/小时
```

## 七、技术选型说明

### 7.1 技术栈选择
```yaml
运行时: Node.js 18+ LTS
理由:
  - 与现有技术栈一致
  - 丰富的生态系统
  - 适合IO密集型应用

Web框架: Express.js
理由:
  - 简单易用
  - 社区成熟
  - 中间件丰富

进程管理: PM2
理由:
  - 自动重启
  - 集群模式
  - 监控日志
```

### 7.2 存储选择
```yaml
JSON文件存储:
理由:
  - 简单直接
  - 无需额外依赖
  - 易于备份迁移
  - 适合读多写少场景

对比数据库:
  - JSON文件: 适合 < 100万条记录
  - PostgreSQL: 适合 > 100万条记录
  - 迁移路径: JSON → PostgreSQL（无感迁移）
```

### 7.3 缓存策略
```yaml
内存缓存:
  - 热点数据缓存
  - 查询结果缓存
  - 缓存过期: 5分钟

缓存穿透防护:
  - 空值缓存
  - 布隆过滤器
```
