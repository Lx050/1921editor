# 飞书通知生成器

## 功能

生成三类团队协作规范的飞书通知消息，确保格式统一、信息完整。

### 1. 数据库变更通知

当后端开发者完成数据库 Migration 后，生成通知消息发送到飞书项目群。

**触发示例：**
```
"生成数据库变更通知"
"我刚完成 Migration，帮我生成飞书通知"
"发送数据库变更通知给团队"
```

**所需信息：**
- 分支名称（如 `feature/user-login`）
- 涉及表名（如 `users`）
- 主要变更描述（如 "users 表新增 email 字段"）
- Migration 文件名（如 `1234567890123-AddUserEmail.ts`）

**通知模板（参考 3.4 节）：**

```text
📢 【数据库变更通知】
分支：feature/user-login
涉及表：users
主要变更：users 表新增 email 字段
Migration 文件：1234567890123-AddUserEmail.ts

⚠️ 请执行：
git pull origin develop
npm run migration:run
```

**使用流程：**
1. 开发者提供 Migration 相关信息
2. 生成符合模板的通知消息
3. 开发者复制到飞书项目群发送

### 2. 接口开发完成通知

当后端接口开发完成后，生成通知消息通知前端。

**触发示例：**
```
"生成接口开发完成通知"
"帮我通知前端接口已完成"
"发送接口开发完成消息"
```

**所需信息：**
- 模块名称（如 "用户管理模块"）
- 接口列表（如 "POST /api/user/login  用户登录"）
- Swagger 地址
  - 优先尝试从项目配置文件（`main.ts`、`.env`）读取后端端口号
  - 如果无法读取，询问用户端口号，生成完整地址（如 `http://localhost:3000/api-docs`）
- 是否已上传架构文档到飞书任务计划表格

**通知模板（参考 8.3 节）：**

```text
📢 【接口开发完成】
模块：用户管理模块
接口：
- POST /api/user/login  用户登录
- GET /api/user/profile 获取用户信息

📎 Swagger 地址：http://localhost:3000/api-docs
📎 架构文档已上传飞书任务计划表格
```

**使用流程：**
1. 后端开发者提供接口信息
2. 生成符合模板的通知消息
3. 开发者复制到飞书项目群发送

### 3. 接口变更通知

当已有接口发生变更时，生成通知消息提醒前端更新。

**触发示例：**
```
"生成接口变更通知"
"帮我通知前端接口有变更"
"发送接口变更消息"
```

**所需信息：**
- 接口路径（如 "POST /api/user/login"）
- 变更内容（如 "新增 remember 参数（可选）"）

**通知模板（参考 8.3 节）：**

```text
⚠️ 【接口变更】
接口：POST /api/user/login
变更：新增 remember 参数（可选）

📎 请重新查看 Swagger 文档
```

**使用流程：**
1. 后端开发者提供变更信息
2. 生成符合模板的通知消息
3. 开发者复制到飞书项目群发送

## 使用场景

**场景 1：数据库变更**
```
开发者："我刚完成 users 表新增 email 字段的 Migration，帮我生成通知"
→ 询问：分支名称？
开发者："feature/user-login"
→ 询问：Migration 文件名？
开发者："1234567890123-AddUserEmail.ts"
→ 生成通知消息
```

**场景 2：接口开发完成**
```
开发者："用户管理模块的接口开发完成了，帮我生成通知"
→ 询问：有哪些接口？
开发者："POST /api/user/login 用户登录，GET /api/user/profile 获取用户信息"
→ 询问：架构文档是否已上传到飞书？
开发者："是的"
→ 生成通知消息
```

**场景 3：接口变更**
```
开发者："login 接口新增了 remember 参数，帮我生成变更通知"
→ 生成通知消息
```

## 信息收集指引

当开发者请求生成通知时，按需询问以下信息：

### 数据库变更通知
1. 分支名称是什么？
2. 涉及哪些表？
3. 主要变更是什么？
4. Migration 文件名是什么？

### 接口开发完成通知
1. 模块名称是什么？
2. 有哪些接口？（格式：HTTP方法 路径 描述）
3. 后端端口号是多少？（如果无法从配置文件读取）
4. 架构文档是否已上传到飞书任务计划表格？

### 接口变更通知
1. 哪个接口发生了变更？（路径）
2. 变更内容是什么？

## 重要提示

1. **通知模板固定**
   - 严格遵循团队协作准则手册的通知模板
   - 保持格式统一，便于团队识别

2. **通知时机**
   - **数据库变更**：Migration 提交后立即通知
   - **接口完成**：接口开发完成并测试通过后通知
   - **接口变更**：接口修改后立即通知

3. **通知渠道**
   - 所有通知发送到飞书项目群
   - 确保相关人员能及时看到

4. **信息完整性**
   - 确保通知包含所有必要信息
   - Migration 文件名、Swagger 地址等细节不可遗漏

5. **相关 Skills 协作**
   - **接口开发完成通知**：通常在使用 `backend-doc-writer` 完成模块文档后生成
